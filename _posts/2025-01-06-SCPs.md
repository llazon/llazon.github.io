# AWS Organizations & Service Control Policies: Patterns That Actually Work

If you're managing more than a handful of AWS accounts, you've probably already discovered that IAM policies alone aren't enough to keep things under control. Service Control Policies (SCPs) are the guardrails that let you sleep at night — they define the maximum permissions available to accounts in your organization, regardless of what IAM policies say.

I've spent a good chunk of time refining our SCP strategy, and I want to share the patterns that have held up in practice.

## Quick Refresher: How SCPs Work

SCPs don't grant permissions. They restrict them. Think of them as a filter that sits above IAM — if an SCP doesn't allow an action, no IAM policy in that account can grant it. They apply to all principals in the account except the root user (though you should never be using root anyway).

Key things to remember:
- SCPs affect member accounts, not the management account
- They're evaluated as an intersection with IAM policies
- A deny in an SCP always wins
- They don't affect service-linked roles

## The OU Structure That Works

Before diving into SCPs, your OU structure matters. Here's a pattern that scales well:

```
Root
├── Security
├── Infrastructure
├── Workloads
│   ├── Production
│   ├── Staging
│   └── Development
├── Sandbox
└── Suspended
```

Each OU gets its own set of SCPs. The key insight: SCPs are inherited, so policies attached to parent OUs apply to all children. Use this to your advantage.

## Pattern 1: Region Restriction

This is usually the first SCP everyone implements, and for good reason. If you only operate in `us-east-1` and `us-west-2`, there's no reason to allow resource creation anywhere else.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyNonApprovedRegions",
      "Effect": "Deny",
      "NotAction": [
        "a4b:*",
        "budgets:*",
        "ce:*",
        "chime:*",
        "cloudfront:*",
        "cur:*",
        "globalaccelerator:*",
        "health:*",
        "iam:*",
        "importexport:*",
        "organizations:*",
        "route53:*",
        "route53domains:*",
        "s3:GetBucketLocation",
        "s3:ListAllMyBuckets",
        "sts:*",
        "support:*",
        "trustedadvisor:*",
        "waf:*"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:RequestedRegion": [
            "us-east-1",
            "us-west-2"
          ]
        }
      }
    }
  ]
}
```

The `NotAction` list is critical — these are global services that don't operate in a specific region. Miss one and you'll break something. I keep this list updated as AWS adds new global services.

## Pattern 2: Prevent Leaving the Organization

Simple but essential. You don't want someone accidentally (or intentionally) detaching an account.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyLeaveOrganization",
      "Effect": "Deny",
      "Action": "organizations:LeaveOrganization",
      "Resource": "*"
    }
  ]
}
```

## Pattern 3: Protect Security Tooling

This one has saved us more than once. Prevent anyone from disabling CloudTrail, GuardDuty, Config, or Security Hub in member accounts.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ProtectSecurityServices",
      "Effect": "Deny",
      "Action": [
        "cloudtrail:StopLogging",
        "cloudtrail:DeleteTrail",
        "guardduty:DeleteDetector",
        "guardduty:DisassociateFromMasterAccount",
        "config:StopConfigurationRecorder",
        "config:DeleteConfigurationRecorder",
        "securityhub:DisableSecurityHub",
        "securityhub:DeleteMembers"
      ],
      "Resource": "*"
    }
  ]
}
```

## Pattern 4: Enforce Encryption

Require encryption on resources that support it. This example covers S3 and EBS:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyUnencryptedS3",
      "Effect": "Deny",
      "Action": "s3:PutObject",
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "s3:x-amz-server-side-encryption": [
            "AES256",
            "aws:kms"
          ]
        },
        "Null": {
          "s3:x-amz-server-side-encryption": "false"
        }
      }
    },
    {
      "Sid": "DenyUnencryptedEBS",
      "Effect": "Deny",
      "Action": "ec2:CreateVolume",
      "Resource": "*",
      "Condition": {
        "Bool": {
          "ec2:Encrypted": "false"
        }
      }
    }
  ]
}
```

## Pattern 5: Sandbox Account Guardrails

Sandbox accounts need freedom to experiment but guardrails to prevent cost explosions:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyExpensiveInstances",
      "Effect": "Deny",
      "Action": "ec2:RunInstances",
      "Resource": "arn:aws:ec2:*:*:instance/*",
      "Condition": {
        "ForAnyValue:StringNotLike": {
          "ec2:InstanceType": [
            "t3.*",
            "t3a.*",
            "t4g.*",
            "m5.large",
            "m5.xlarge"
          ]
        }
      }
    },
    {
      "Sid": "DenyRDSLargeInstances",
      "Effect": "Deny",
      "Action": "rds:CreateDBInstance",
      "Resource": "*",
      "Condition": {
        "ForAnyValue:StringNotLike": {
          "rds:DatabaseClass": [
            "db.t3.*",
            "db.t4g.*"
          ]
        }
      }
    }
  ]
}
```

## Testing SCPs Without Breaking Everything

This is where people get burned. A bad SCP can lock you out of an account. Here's my approach:

1. Start with a single test account in a dedicated OU
2. Attach the SCP to that OU only
3. Test with a non-admin role to verify the deny works
4. Test with your admin role to verify you haven't broken admin access
5. Check that service-linked roles still function
6. Gradually roll out to broader OUs

Use the IAM Policy Simulator and `aws sts get-caller-identity` combined with test API calls to validate behavior before rolling out.

## SCP Size Limits and How to Deal With Them

SCPs have a 5,120 character limit per policy, and you can attach a maximum of 5 SCPs per OU. That's not a lot of room. Some tips:

- Use `NotAction` instead of listing every action you want to allow
- Combine related denies into a single SCP
- Use condition keys to reduce the number of statements
- If you're hitting limits, it might be a sign your OU structure needs refactoring

## Lessons Learned

- Always test in isolation first. Always.
- Keep a "break glass" procedure documented for when an SCP blocks something critical
- Version control your SCPs — treat them like code
- Review SCPs quarterly; AWS adds new services constantly
- The management account is exempt from SCPs, which is both a feature and a risk — lock it down with IAM instead

SCPs are one of those things that feel tedious to set up but pay dividends every single day. Get them right and your multi-account environment becomes dramatically easier to manage and secure.
